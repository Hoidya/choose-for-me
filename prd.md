需求文档（PRD）

项目名（暂定）：Choose For Me / 决定吧
产品类型：解决选择困难的轻游戏化网站
部署方式：Vercel（前端）+ Supabase（数据库）

⸻

一、产品背景 & 目标

1. 背景

选择困难是一个高频、低风险、强共鸣的问题场景：
• 吃什么
• 去哪
• 谁买单
• 做 / 不做
• 选 A 还是 B

传统做法要么是工具型（冷），要么是游戏型（幼稚）。
本产品目标是：用“游戏机制”包装“现实决策”，让选择看起来像命运，而不是责任。

⸻

2. 核心目标
   • 降低决策成本
   • 把“做决定”转化为“玩一把”
   • 支持可复用的选择模板
   • 不追求复杂算法，追求爽感和确定性

⸻

二、核心用户画像

目标用户
• 18–35 岁
• 对「选择 / 决定 / 命运 / 随机性」不排斥
• 日常高频使用工具类产品
• 不想被教育，只想“快点决定”

使用场景
• 晚上吃什么
• 朋友聚会谁付钱
• 旅行目的地
• 要不要联系某个人
• 今天做什么事

⸻

三、产品结构（信息架构）

首页
├─ 新建一个选择
│ ├─ 选择类型
│ │ ├─ 转盘
│ │ ├─ 骰子

⸻

四、核心功能需求

1. 新建选择（核心）

功能说明
用户创建一个“问题 + 选项”的选择任务。

输入项
• 问题（必填）
• 示例：
• 今晚吃什么？
• 去哪旅行？
• 谁买单？
• 选择方式（必填）
• 转盘
• 骰子
• 抽卡
• 选项列表（2 个起）
• 文本输入
• 例如：
• 火锅 / 日料 / 麦当劳
• 北京 / 东京 / 曼谷

⸻

2. 游戏机制设计

A. 转盘
• 圆盘平均分配选项
• 点击「开始」旋转
• 随机停在某一项
• 显示结果（放大 + 动画）

B. 骰子
• 每个选项对应一个点数
• 掷骰动画
• 显示点数 → 映射选项

C. 抽卡
• 所有选项作为卡牌背面
• 点击抽一张
• 翻牌动画
• 显示结果

📌 规则原则：
公平、不可手动干预、结果唯一

⸻

3. 结果展示
   • 大字号显示最终选项
   • 配一句“命运式文案”（随机）
   • “就这样了。”
   • “宇宙已经替你决定。”
   • “不要反悔。”
   • 一个按钮：
   • 「再来一次」
   • 「保存这个选择」

⸻

4. 保存 & 复用（Supabase）

保存内容
• question（问题）
• type（转盘 / 骰子 / 抽卡）
• options（数组）
• created_at
• user_id（匿名或登录）

功能
• 用户可保存常用选择
• 在「我的选择」中一键重玩
• 可编辑 / 删除

📌 初期可 不做复杂登录
• 用匿名 session
• 或 Supabase anonymous auth

⸻

五、非核心但加分功能（可延后）
• 分享链接（把选择发给朋友一起看结果）
• 历史记录（你曾被命运选中过什么）
• 选择概率自定义（进阶）

⸻

六、技术方案（偏 MVP）

前端
• Framework：Next.js / React
• UI：简单、干净、不幼稚
• 动画：CSS / Framer Motion（轻量）

后端 / 数据
• Supabase
• Table: choices
• 字段：
• id
• question
• type
• options (json)
• created_at
• user_id

部署
• Vercel
• 环境变量：
• Supabase URL
• Supabase anon key

⸻

七、产品风格 & 氛围

关键词：
• 冷静
• 命运感
• 不卖萌
• 不教育
• 不“心理学味”

视觉方向：
• 黑白 / 中性色
• 大字号
• 明确反馈
• 少 icon，多留白

⸻

八、MVP 验收标准

上线版本必须满足：
• ✅ 能创建问题
• ✅ 能选择 3 种游戏之一
• ✅ 能随机给出结果
• ✅ 能保存并再次使用
• ✅ 能在 Vercel 访问

不要求：
• 登录体系完善
• 商业化
• 复杂推荐

九、一句话产品定义（以后可用）

这是一个把“做决定”变成“交给命运”的网站。
